What Is AWS Cognito?

AWS Cognito is Amazon's identity service for web and mobile apps. It handles sign-up, sign-in, user profiles, token generation, and secure access so our application can focus on business logic.

User Authentication JWT Tokens Hosted UI SSO Ready
User Pool Stores users, passwords, profile attributes, groups, and sign-in policies in AWS.
Hosted UI Provides a ready-made login page so we do not need to build authentication screens from scratch.
OAuth2 + OIDC Supports authorization code flow for browsers and standards-based login across modern apps.
API Security Issues access tokens that backends can validate before serving protected endpoints.
Why teams use it It reduces custom auth code, centralizes user identity, and integrates cleanly with backend APIs.
Common use cases Customer portals, admin dashboards, mobile app login, single sign-on, and API authorization.
What it returns ID tokens for profile data, access tokens for APIs, and refresh tokens for longer sessions.
Fit for this repo This project uses Cognito Hosted UI for browser login and JWT validation for Bearer-token API access.

Architecture Of This Project

This application combines Spring Security OAuth2 Login and Resource Server support so the same backend can handle browser-based Cognito login and direct JWT-based API authentication.

Browser / API Client Calls `/api/orders` or `/public/health`.
→
Spring Security `SecurityConfig` protects `/api/**` and allows `/public/**`.
Cognito Hosted UI Used when the request is from a browser without an authenticated session.
←→
AWS Cognito User Pool Authenticates the user and issues OAuth2 / OIDC tokens.
JWT Validation Resource Server validates Bearer tokens using the configured issuer URI.
→
`OrderController` Reads `email`, `cognito:username`, or `sub` and responds with user-specific data.
1. Public access `/public/health` is open and returns a simple status response.
2. Browser login flow Unauthenticated HTML requests are redirected to `/oauth2/authorization/cognito`, then to Cognito Hosted UI.
3. Token exchange After login, Spring receives the authorization code and creates the authenticated user session.
4. API token flow Clients can also send `Authorization: Bearer <token>` and access the same protected API.
5. Logout `CognitoLogoutSuccessHandler` redirects to Cognito `/logout` so both local and Hosted UI sessions are cleared.
Key config: `issuer-uri`, OAuth2 client registration, JWT resource server, custom logout handler Project reference: Spring Boot + Spring Security + AWS Cognito
Slide 3

How Hosted UI Auth Works Behind The Scenes

Cognito Hosted UI handles the login screen, but Spring Security coordinates the redirects, exchanges the authorization code for tokens, and creates the local authenticated session.

1. Protected request The browser calls `/api/orders` without an existing session.
->
2. Spring entry point `SecurityConfig` redirects HTML requests to `/oauth2/authorization/cognito`.
3. Hosted UI redirect Spring builds the OAuth2 authorize request and sends the browser to Cognito Hosted UI.
->
4. User login The user signs in on Cognito, not inside the Spring Boot application.
5. Callback with code Cognito redirects to `/login/oauth2/code/cognito?code=...&state=...`.
->
6. Backend token exchange Spring uses the client ID and secret to exchange the code for ID, access, and refresh tokens.
7. Local session is created Spring validates the tokens, builds an `OidcUser`, and stores the authenticated session in the app.
8. Controller reads claims `OrderController` can resolve `email`, `preferred_username`, `cognito:username`, or `sub` from the authenticated principal.
9. API mode also works For Bearer-token requests, Spring skips redirects and validates the JWT directly using Cognito issuer metadata.
10. Logout has two parts The app clears the local session, then `CognitoLogoutSuccessHandler` redirects to Cognito `/logout` to clear the Hosted UI session too.
Main idea The browser sees redirects, the backend handles the secure token exchange, and Cognito remains the identity provider.
Flow: browser redirect, authorization code callback, backend token exchange, local Spring session Based on this repo's `oauth2Login` and resource server configuration
